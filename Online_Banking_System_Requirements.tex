\section{Introduction}

\subsection{System Overview}
The Online Banking System (OBS) is a central web application designed to handle day-to-day financial operations for retail and corporate bank customers. The system serves as the primary digital interface, connecting the bank's core banking database infrastructure with user-facing devices. It provides integrated interfaces for account monitoring, domestic funds transfer, utility bill payments, real-time alerts, and self-service account management.

\subsection{System Scope \& Core Operational Boundaries}
The scope of the OBS covers the development of five core application modules:
\begin{itemize}
    \item \textbf{Secure Authentication:} Customer signup, secure login, multi-factor authentication (MFA) via SMS or authenticator apps, and security log recording.
    \item \textbf{Account Management:} Real-time balance checks, interest tracking, profile management, and account ledger updates.
    \item \textbf{Funds Transfer System:} Instant and scheduled transfers, supporting intra-bank transactions, ACH transfers, and high-value wire transfers (RTGS).
    \item \textbf{Utility Billing \& Automated Payments:} Direct integration with external billing services (such as electricity, water, and telecom providers) and automatic recurring payment configurations.
    \item \textbf{Reporting \& PDF Statements:} On-demand generation and downloading of transaction histories in a structured, standard PDF format.
\end{itemize}

\subsection{Target User Base \& System Actors}
The OBS coordinates operations among five distinct user roles and external systems:
\begin{enumerate}
    \item \textbf{Individual Customer (Primary Actor):} A retail customer who uses the application for personal accounts management, transfers, bill payments, and statement retrieval.
    \item \textbf{Corporate Customer (Primary Actor):} A business user executing larger, bulk transfers (ACH/RTGS) that require multi-step authorization workflows and corporate reporting.
    \item \textbf{System Administrator (Secondary Actor):} An IT administrator responsible for platform configuration, managing user roles, checking backups, and reviewing audit logs.
    \item \textbf{Bank Teller (Secondary Actor):} A bank employee who handles customer disputes, verifies manual transfer overrides, and reviews transaction logs.
    \item \textbf{Third-Party APIs (External Actors):} Clearing networks (Shetab, RTGS, ACH), external SMS gateway services, and government identity verification databases.
\end{enumerate}

\section{Functional Requirements}
Functional requirements define the core operations, transactions, and services that the Online Banking System must perform. To ensure the specifications are accessible to business stakeholders while remaining technically rigorous for developers and testers, each requirement is presented in a dual format:
\begin{itemize}
    \item \textbf{User Requirements:} Written in non-technical natural language to describe the features from a customer's perspective.
    \item \textbf{System Requirements:} Detailed engineering specifications that use strict "**shall**" terminology to define absolute system obligations, protocols, and technical constraints.
\end{itemize}

\subsection{System Operational Context Model}
The System Operational Context Model establishes the boundary of the Online Banking System (OBS), categorizing the components into internal subsystems and external interfaces. The central OBS encompasses eight core internal subsystems responsible for user lifecycle and financial data transactions. The Authentication \& MFA subsystem manages secure sessions, while the Account Management and Funds Transfer subsystems handle direct ledger actions. Reporting \& Statement, Notification, Audit \& Logging, and Backup \& Recovery subsystems provide critical support functions ensuring reliability, transparency, and compliance.

Surrounding the central system are the external entities that interact directly with the OBS. Human actors—including Individual Customers, Corporate Customers, Bank Tellers, and System Administrators—interface through browser and mobile client terminals. External automated systems interact via secure API channels: the Identity Verification Service validates signup credentials, the SMS Gateway handles out-of-band notifications, Utility Providers exchange billing tokens, and the Shetab/ACH/RTGS clearing house settles inter-bank transfers. This context model ensures a robust definition of system interfaces, security perimeters, and operational dependencies.

\subsection{UML Activity Diagrams}
UML Activity Diagrams map the dynamic control flows and decision behaviors within the Online Banking System. The diagrams below model the sequential actions, choices, and terminal states for each of the eight primary use cases (UC1 to UC8).

\subsubsection{AD1: Register Account \& AD2: Login with MFA}
The Register Account activity diagram (AD1) models the sequential control flow of customer onboarding. The process begins with the inputs of user profile details, which are verified on the server side against format and cryptographic constraints. If validation fails, an error is rendered, prompting the user to correct the inputs. Upon successful validation, the system initiates the MFA setup by issuing an out-of-band OTP. A second decision gateway verifies the OTP entry; successful validation triggers the creation of the database records and initializes the MFA key, concluding the registration process.

The Login with MFA workflow (AD2) outlines the session authentication lifecycle. The system prompts for primary user credentials and validates the hashed password against database tables. If the password matches, the login flow moves to secondary challenge verification, dispatching a dynamic token. If the user successfully inputs the matching OTP within the time window, the system generates a secure session token and redirects the client to their customized dashboard, safeguarding the system from credential abuse.

\subsubsection{AD3: View Balance \& AD4: Transfer Funds}
The View Balance activity diagram (AD3) traces a lightweight read-only operation. The client requests a balance inquiry, passing their authenticated session token. The application server validates the token structure and queries the core account ledger. If database connectivity or ledger retrieval fails, the system renders an error message. Otherwise, it compiles the ledger balances and updates the user interface, reflecting real-time account values.

The Funds Transfer diagram (AD4) manages a highly critical, multi-stage state modification. The user inputs transaction details, including the target IBAN and amount. The system first checks that the source account holds sufficient funds to cover the transfer amount and transaction fees. If valid, the system validates the format and active status of the destination IBAN. Once these checks clear, the system prompts the user for authentication confirmation, executes the debit and credit actions atomically under database transaction locks, and invokes the SMS gateway API to dispatch receipts to both parties.

The system queries the external provider's API to fetch outstanding balances. If a balance is due, the application checks checking ledger availability, debits the account, transmits the payment approval back to the utility gateway, and prints a settlement confirmation.

The Generate Statement workflow (AD6) details the document compilation engine's flow. The user specifies filter parameters (such as transaction type and date range). The server validates the parameters, ensuring dates are chronological. It then executes database query operations to fetch matching records. If records exist, the server formats the transaction history, generates a secure PDF, and streams the document to the browser, wrapping up a clean read operation.

\subsubsection{AD7: Manage Backups \& AD8: Review Transaction Logs}
The Manage Backups diagram (AD7) models the background administrative disaster recovery flow. Activated by an automated cron scheduler, the backup subsystem generates an encrypted snapshot of the database volumes. The system checks the file integrity against checksums. If integrity is confirmed, the snapshot is uploaded to offsite cloud storage. Failures trigger immediate notifications to the IT administration panel, ensuring data reliability.

The Review Transaction Logs diagram (AD8) handles compliance and auditing workflows. Tellers and Administrators access the audit dashboard to search security logs using filters. The query compiles matching records from the immutable audit database. If logs are found, they are formatted and displayed. Security flags or critical violations prompt immediate escalations, ensuring real-time auditing and regulatory tracking.
